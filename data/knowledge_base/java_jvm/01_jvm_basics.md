# Java JVM 核心

## 1. JVM 内存模型

### 运行时数据区
```
JVM 内存结构
├── 线程共享
│   ├── 堆 (Heap)           — 对象实例、数组（GC 主要区域）
│   └── 方法区 (Method Area) — 类信息、常量池、静态变量（JDK8+ 元空间替代永久代）
└── 线程私有
    ├── 程序计数器 (PC)       — 当前线程执行的字节码行号
    ├── 虚拟机栈 (VM Stack)   — 方法调用：栈帧（局部变量表/操作数栈/动态链接）
    └── 本地方法栈 (Native)   — Native 方法
```

### 堆内存细分（分代模型）
```
堆 = 新生代(1/3) + 老年代(2/3)
新生代 = Eden(8/10) + S0(1/10) + S1(1/10)
```
- 对象优先在 Eden 分配
- Minor GC 存活 → Survivor（S0 ↔ S1 来回，够 15 次晋升老年代）
- 大对象直接进老年代（-XX:PretenureSizeThreshold）

### 元空间 (Metaspace) vs 永久代 (PermGen)
- JDK8 用 Metaspace 替代 PermGen
- Metaspace 使用**本地内存**，不再有 `java.lang.OutOfMemoryError: PermGen`
- 参数：`-XX:MetaspaceSize` / `-XX:MaxMetaspaceSize`

### 常见面试题
1. **堆和栈的区别？** 堆存对象（共享），栈存方法调用和局部变量（私有）
2. **JDK8 为什么移除永久代？** 解决 PermGen OOM，Metaspace 用本地内存更灵活
3. **一个对象从创建到回收的完整路径？** Eden → S0/S1 → 老年代 → GC 回收

## 2. 垃圾回收 (GC)

### 判断对象存活
- **引用计数法**：有循环引用问题，Java 不用
- **可达性分析**：从 GC Roots 出发，不可达的对象判定为可回收
  - GC Roots：虚拟机栈引用、静态变量、常量池引用、JNI 引用

### 垃圾回收算法
| 算法 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| 标记-清除 | 标记存活 → 清除未标记 | 简单 | 内存碎片 |
| 标记-整理 | 标记存活 → 向一端移动 | 无碎片 | 耗时 |
| 复制 | 分两块，存活复制到另一块 | 效率高 | 浪费一半空间 |
| 分代收集 | 新生代用复制，老年代用标记-整理 | 综合最优 | — |

### 主流 GC 收集器
| 收集器 | 区域 | 算法 | 特点 |
|--------|------|------|------|
| Serial | 新生代 | 复制 | 单线程，Client 模式默认 |
| ParNew | 新生代 | 复制 | Serial 多线程版 |
| Parallel Scavenge | 新生代 | 复制 | 吞吐量优先 |
| CMS | 老年代 | 标记-清除 | 低停顿，已废弃 |
| G1 | 全堆 | 分区 | 平衡吞吐量和延迟（JDK9+ 默认） |
| ZGC | 全堆 | 染色指针 | 超低延迟（<1ms 暂停） |

### 常见面试题
1. **Minor GC / Major GC / Full GC 的区别？** Minor→新生代, Major→老年代, Full→整个堆+方法区
2. **G1 相比 CMS 的优势？** 可预测的停顿时间、不会产生内存碎片、并发标记
3. **GC 调优思路？** 先定目标（吞吐/延迟）→ 选收集器 → 调堆大小 → 调分代比例
